# Phase 8 Report

## Summary

Phase 8 copied the completed Phase 7 app forward and refined the sensor provider pipeline and Sensors UI inside:

`PHASE_8_SENSOR_PROVIDER_FIXES_AND_UI_REFINEMENT`

This phase is telemetry, diagnostics, and UI only. It does not enable MSI Afterburner profile launching, NVIDIA writes, fan/EC writes, service control, startup entries, scheduled tasks, registry writes, automatic game apply, or automatic CPU/GPU tuning apply.

## What Changed From Phase 7

- Added explicit sensor validity states: valid, unavailable, stale_zero, invalid_value, unsupported, hidden_by_firmware, no_provider, idle_no_capture, not_started, and read_error.
- Fixed CPU partial-provider behavior so valid load/voltage can be shown while invalid temperature and stale-zero power/clock remain unavailable.
- Added LibreHardwareMonitor multi-sample read support with sample values, nonzero counts, and stale-zero flags.
- Added a read-only CPU telemetry provider adapter for LHM, Windows counters, ACPI diagnostic status, and HWiNFO detection planning.
- Improved GPU classification for GpuNvidia and GpuAmd hardware types.
- Added validity, validity reason, provider, and subcategory columns to All Sensors.
- Added CPU provider diagnostics export to app-side JSON only.
- Updated Dashboard and Settings/Safety to show Phase 8 sensor/provider status.
- Updated root/direct launch behavior to use Phase 8 assets.

## CPU Sensor Issue Diagnosis

For the provided live snapshot pattern:

- CPU load is valid.
- CPU temperature is unavailable because the only live candidate, Core (Tctl/Tdie), reports 0 C and is rejected as invalid.
- CPU power is treated as stale-zero/unavailable when it reports 0 W while CPU load is nonzero.
- CPU clock is treated as stale-zero/unavailable when it reports 0 MHz while CPU load is nonzero.
- CPU voltage is valid when a reasonable VID voltage is exposed.

This makes the app show useful CPU telemetry without pretending that 0 W or 0 MHz are real running values.

## GPU Classification

Phase 8 now classifies GpuNvidia, GpuAmd, hardware types starting with Gpu, NVIDIA/RTX/GeForce text, AMD Radeon text, and GPU text into GPU categories/subcategories. GPU Memory Junction 255 C is marked invalid_value.

## UI Refinements

- PresentMon no CSV/no capture appears as idle rather than a warning-like error.
- Provider status is compact and source-aware.
- Raw sensor explorer remains complete, but now explains validity directly.

## Known Issues

- CPU temperature may remain unavailable on the AMD Ryzen AI platform if LibreHardwareMonitor only exposes Core (Tctl/Tdie) as 0 C.
- CPU power and clock may remain stale-zero through LHM on this platform.
- HWiNFO shared memory is detected/planned only; no HWiNFO parser is enabled in Phase 8.
- Windows counter and ACPI probes are not run automatically during normal UI rendering.

## Recommended Phase 9

Focus on HWiNFO shared-memory detection/readout if available, Windows counter fallback for CPU frequency/load, manual sensor pinning improvements, sensor naming overrides, and exportable telemetry diagnostics. Do not enable MSI, NVIDIA, fan, EC, service, startup, scheduled task, registry, or automatic game-apply writes.
